DNA

- Union of several data: - Integrity Code - Defining Validation Rules - Properties - Arbitrary configuration data - Network Seed - Glorified property to uniquify equivalent looking networks - Is identified by the hash of its components

Zomes

- Source code modules that form a DNA - 2 types - Integrity zomes - Included in the hash function to compute the DNA hash. - Can't be updated, if you change them you have created a new DNA. - They declare: - Entry and link types - Genesis self-check callback - Will run when you are installing an application - If it fails, the installation process fails - Validation rules - Coordinator zomes - Not included in the hash function to compute the DNA hash - Can be updated, you can change them without changing the DNA hash - They declare: - CRUD functions - Other callbacks (eg. init)

DNA manifest & bundle

- Manifest declares: - Human readable name - Zomes that will be included in the DNA - Their names - The location of their compiled source code - Provisional properties - Provisional UID - Manifest is packaged into a DNA bundle

hApp manifest & bundle

- Manifest declares: - Human readable name - DNAs that will be included in the hApp - Their role ids (names) - The location of their DNA bundle - Provisionging strategy: whether to create a new cell on install - Manifest is packaged into a hApp bundle - The conductor can directly install hApp bundles

Runtime Architecture

Cell

- Unique per-agent instantiation of a DNA - Is identified by the union of an agent ID, and a DNA hash - Has access to: - A private data storage: the source chain - A shared and public data storage: the DHT - Only accessible via permissioned zome function calls: - By an external process connected to the Conductor (eg. GUIs) - By other cells from its network (with the same DNA hash) - By other cells in the same conductor - Can receive these parameters when it is instantiated: - Parameters that change the resulting DNA hash: - Properties: change the behaviour of the DNA - UID: arbitrary string - Membrane proof: may be required to instantiate the Cell

Merkle Trees

- Merkle tree are DAGs in which new objects can reference other objects by its hash - A merkle chain is a merkle tree that has no branches - It becomes impossible to change one object in the list without changing all of the following objects in the chain, making a tampering attempt impossible

Source Chain (I)

- A Source Chain is a Holochain specific merkle chain - Each Cell controls its own local source chain - The chain contains all the actions that a Cell has taken, announce within the scope of the network - Each action contains the hash of the previous action, making the source chain a merkle chain

Source Chain (II)

- Each action is signed by its author - Each action may be accompanied by some entry, defined by the application - The action is public in the DHT for the Cell - The entry may be private or public, as defined by the application - Each action contains: - Author's public key - Timestamp - Hash of the previous action - Entry hash, if it's accompanied by an entry - A record is a union of an action and its entry - Extra credit: This merkle chain is used to preserve privacy and data integrity. If any action was private, then agents would be able to fork their chain without nobody noticing.

Source Chain (III)

- At boot time, each source chain will be instantiated with 3 records: - DNA: includes the DNA hash. - AgentValidationPackage: joining proof that satisfies the requirements that the DNA rules declare, making the 3rd record valid - Cryptographic signatures: signing and verifying functions. - Create record of its own agent public key. Analogy: your phone number in the network.

Hash-Table

- Key-value store that allocates a bigger chunk of memory than necessary - To insert a value, we hash the **key**, and put the content in the position of memory that corresponds to that hash - Constant lookup and write time

Cryptographic Hash-Table

- Key-value store where the key is the cryptographic hash of the **value** - After lookup, you can hash the data again and compare it to the querying hash to verify that you are not being lied to

Distributed cryptographic Hash-Table (DHTs)

- Cryptographic Hash-Table where the hash-table is spread accross many nodes - Each node can make a get request with the hash of some value, and the network will return the value itself

Holochain's DHT (I)

- Two type of network queries: - Get a value from its hash - Get the metadata attached to a given hash - Guarantees: - - 1st guarantee: data integrity: - You can verify that the data is correct since you can hash it again like in any cryptographic hash-table. - Actions are guaranteed to come from its author because they are signed. - 2nd guarantee: the DHT is resilient: - Censorship resistance : it would be impractical for the network to selectively censor a piece of data in the hash-table. - Availability: pieces of data will likely still be in the network even in outage scenarios (pieces of data are likely to be held in the network).

Holochain's DHT (I)

- 3rd guarantee: - Consistency: - The pieces of data cannot contradict one another. - Eventual consistency: - CRDTs: - There is no universal sequence of order of events valid (as per relativity), and consequently order of events in distributed systems is hard! - Data structure that only implements commutative operations. - And it implements a state machine that reduces a state from a set of operations. - Any agent given the same operations will arrive at the same end state. - The DHT is a big CRDT. - The data state converges to being free of internal contradiction. - There is a guarantee that operations will get to you eventually. - Reveals its own temporary inconsistencies. - Holochain provides a way to resolve the temporary inconsistencies.

hApp (Holochain App)

- An assemblage of instantiated cells, among which an agent id is shared - Each cell has its own role ID - Eg: in a file storage app, you may have two cells installed: - “main”: the main DHT that everyone that installs the app joins - “file_storage_provider”: only joined by high-storage nodes - Can depend on cells from other hApps

Conductor: the Holochain node

- Local process that houses all of your running happs - Offers a sandboxed execution environment with limited capacities to each cell - There is no access to APIs to the underlying OS, eg: - Filesystem - HTTP network calls - Provides a mechanism of communication across cells - Manages all network communications - Exposes two types of interface, accessible via websockets: - App Interface - Receives zome function calls - Can send push signals to subscribers - Admin Interface - Cannot be accessed from an app - Receives admin commands to manage apps - Install App - Enable App - Uninstall App - List Apps - ...
